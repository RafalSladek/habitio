Option 2: Google Drive as a database (app-only, no backend) This is the most interesting option for your case. The Google Drive API allows a web app to read/write a small JSON file in the user's own Drive — using a special "appDataFolder" that's invisible to the user but tied to your app. What changes: you add the Google Identity Services SDK (a single script tag), configure a Google Cloud project with Drive API enabled, and add ~100 lines of sync logic. The HTML file calls gapi to authenticate with OAuth, then reads/writes a habits.json file in your Drive's hidden app folder. Tradeoff: requires a Google Cloud project (free), an OAuth consent screen, and a client ID hardcoded in the HTML. Still a standalone HTML file. Works across devices. User signs in once per session. Option 3: Firebase/Supabase backend A small cloud database (Firestore or Supabase) where data syncs in real-time. You'd add the Firebase JS SDK (single script import), authenticate with Google Sign-In, and read/write to a Firestore document. What changes: ~80 lines of sync code, a Firebase project (free tier), and a firebaseConfig object in the HTML. Still a standalone HTML file. Gives you real-time sync across multiple devices simultaneously. Tradeoff: you're now dependent on a third-party service. But Firebase free tier is more than enough for personal use. Option 4: Your own server + a JSON file Since you mentioned you have a server — the simplest backend would be a tiny API (3 endpoints: GET /data, PUT /data, GET /health) that reads/writes a single JSON file on disk. Could be 30 lines of Node or Python. What changes: add fetch calls in the HTML to sync on load and on every save. Add basic auth (even just a hardcoded token since it's personal). Tradeoff: you maintain a server process, but you fully own the data with no third-party dependency.
